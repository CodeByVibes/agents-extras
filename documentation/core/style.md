# Documentation Style Guide

This style guide dictates the tone, language, and voice used in our documentation.

## Tone and Voice
- **Concise and Direct:** Get straight to the point. Avoid fluff or filler words.
- **Developer-Focused:** Use industry-standard terminology correctly. Avoid marketing speak.
- **Active Voice:** Write in the active voice. (e.g., "The server handles requests," NOT "Requests are handled by the server.")
- **Second Person:** Address the reader directly using "you" when writing guides or tutorials.

## Vocabulary and Grammar
- **Capitalization:** Use sentence case for headers (e.g., "Setting up the database," NOT "Setting Up The Database"), except for proper nouns and product names.
- **Pronouns:** Use gender-neutral pronouns (they/them) when referring to a hypothetical user.
- **Formatting:** 
  - Use `inline code formatting` for file names, paths, variable names, and commands.
  - **Bold** UI elements (e.g., Click the **Submit** button).

## Code Examples
- Code examples must be functional and self-contained when possible.
- Avoid using meta-syntactic variables like `foo` and `bar`. Use realistic variables like `user`, `order_id`, etc.
- Include comments inside code examples to explain non-obvious logic.
